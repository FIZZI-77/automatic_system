# Auth Service

## Что это за сервис

Auth Service — это отдельный инфраструктурный сервис, который отвечает за аутентификацию пользователей, выпуск токенов доступа, управление сессиями и связанные с этим операции безопасности. Он не содержит бизнес-логики предметной области и не знает ничего о заявках, маршрутах, бригадах или других сущностях системы. Его задача — быть единым центром identity и access management.

Сервис работает по gRPC и используется внутренними компонентами системы, в первую очередь API Gateway. Именно через него проходят операции регистрации, входа, обновления токенов, выхода из системы, подтверждения почты и восстановления пароля.

## Из чего состоит сервис

Сервис разделен на несколько уровней.

Handler layer принимает gRPC-запросы, преобразует protobuf-сообщения во внутренние input-модели, парсит UUID и передает управление в service layer. Обратно handler собирает protobuf-ответы и возвращает их клиенту.

Service layer содержит основную бизнес-логику. Здесь выполняются проверки входных данных, поиск пользователей, проверка пароля, создание сессий, генерация токенов, ротация refresh token, подтверждение почты, сброс пароля и отправка email через mail service.

Repository layer работает только с PostgreSQL. Он читает и изменяет данные в таблицах пользователей, сессий, refresh token, ролей и одноразовых токенов. Для составных сценариев, где нужно менять несколько сущностей атомарно, используется отдельный транзакционный репозиторий.

Mail service отвечает за отправку писем подтверждения почты и восстановления пароля. Он не решает бизнес-задачи, а только формирует и отправляет сообщение по SMTP.

## Основные сущности

В сервисе используются несколько ключевых сущностей.

Пользователь хранит базовую учетную информацию: email, username, password hash, флаги активности и подтверждения почты, а также даты создания и обновления записи.

Сессия отражает конкретный вход пользователя в систему. У сессии есть свой ID, привязка к пользователю, данные о клиенте, IP, user-agent, срок жизни и признаки отзыва.

Refresh token хранится в базе только в виде hash. Сам токен отдается клиенту в raw-виде, а затем при refresh сравнивается по hash. Это снижает риски при утечке базы.

Роли и связи пользователь–роль используются для формирования auth-информации и записи ролей в JWT claims.

Одноразовые токены используются для подтверждения почты и сброса пароля. Они также хранятся в базе в виде hash, имеют тип, срок действия и отметку об использовании.

## Как работает аутентификация

После успешного логина сервис выдает две сущности: access token и refresh token.

Access token — это JWT с коротким временем жизни. Он не хранится в базе и нужен для доступа к системе. В него кладутся claims, которые описывают пользователя и контекст входа: идентификатор пользователя, идентификатор сессии, роли и стандартные поля JWT, такие как срок действия, время выдачи, издатель и целевая аудитория.

Refresh token живет дольше и используется для получения новой пары токенов. В базе хранится только его hash. Каждый refresh token привязан к конкретной сессии и пользователю.

Для подписи JWT используется RSA-приватный ключ. Публичная часть ключа отдается через JWKS и используется API Gateway для проверки подписи.

## Как работает регистрация

При регистрации сервис получает email, username и пароль. Данные валидируются на уровне input-модели. Затем сервис проверяет, существует ли уже пользователь с таким email. Если нет, пароль хешируется через bcrypt, после чего в базе создается новый пользователь.

На этапе регистрации сессия не создается и токены не выдаются. Регистрация создает только учетную запись. Дальше пользователь может либо сразу авторизоваться, либо пройти подтверждение почты, если это требуется политикой безопасности.

## Как работает логин

При логине сервис получает email, пароль и служебную информацию о клиенте: client_id, IP и user-agent. Сервис находит пользователя по email, проверяет, что учетная запись активна, и сравнивает введенный пароль с сохраненным password hash через bcrypt.

После успешной проверки создается новая сессия. Затем сервис получает роли пользователя, генерирует access token и refresh token, сохраняет hash refresh token в базе и возвращает клиенту access token, refresh token, session_id, token_type и сроки действия токенов.

## Как работает refresh

Когда access token истекает, клиент отправляет refresh token. Сервис не ищет raw token напрямую. Сначала он считает hash пришедшего значения, затем ищет соответствующую запись в таблице refresh token.

После нахождения записи сервис проверяет, что токен существует, не истек, не был заменен новым токеном и связан с действующей сессией. Также проверяется сама сессия: существует ли она, не отозвана ли и не истекла ли.

Если все проверки проходят, сервис генерирует новый access token и новый refresh token. Старый refresh token в транзакции помечается как использованный и отозванный, а новый сохраняется в базе. Это называется refresh token rotation. Такой подход повышает безопасность и позволяет отслеживать повторное использование устаревших токенов.

## Как работает logout

Logout завершает одну конкретную сессию. Сервис получает session_id, проверяет, что сессия существует, и в рамках транзакции отзывает саму сессию и все refresh token, привязанные к ней.

Сущности не удаляются физически. Вместо этого используется soft revoke: у сессии и токенов выставляются признаки отзыва и время отзыва. Это позволяет сохранять историю и упрощает аудит.

## Как работает logout all

Logout all завершает все активные сессии пользователя. После проверки пользователя сервис в транзакции отзывает все его сессии и все refresh token, связанные с этим пользователем. В ответ возвращается количество инвалидированных сессий.

## Как работает смена пароля

Change password используется, когда пользователь авторизован и знает текущий пароль. Сервис получает user_id, session_id, старый пароль, новый пароль и флаг, определяющий, нужно ли завершить только текущую сессию или все сессии.

Сначала сервис проверяет существование пользователя и правильность старого пароля. Затем новый пароль хешируется через bcrypt. После этого в транзакции обновляется password hash и, в зависимости от выбранного режима, либо отзывается только текущая сессия и ее refresh token, либо отзываются все сессии пользователя и все его refresh token.

Таким образом, смена пароля не просто обновляет пароль, но и сразу пересобирает безопасное состояние всех active sessions.

## Как работает подтверждение почты

Подтверждение почты разбито на два действия.

Первый шаг — отправка verification email. Сервис получает пользователя, проверяет, что почта еще не подтверждена, инвалидирует все старые неиспользованные verification token этого типа, создает новый одноразовый токен, сохраняет его hash в базе и отправляет пользователю письмо со ссылкой.

Второй шаг — verify email. Пользователь переходит по ссылке, токен попадает в сервис, сервис считает его hash, находит соответствующую запись в таблице одноразовых токенов, проверяет, что токен не истек и не был использован, после чего помечает email пользователя как подтвержденный и сам токен — как использованный.

## Как работает восстановление пароля

Восстановление пароля тоже состоит из двух этапов.

Сначала пользователь отправляет запрос на reset password по email. Сервис находит пользователя, инвалидирует старые неиспользованные reset token этого типа, создает новый одноразовый токен, сохраняет его hash и отправляет письмо со ссылкой на сброс пароля.

Затем пользователь открывает ссылку и отправляет reset token вместе с новым паролем. Сервис находит токен по hash, проверяет, что он валиден и не использован, хеширует новый пароль и в транзакции обновляет password hash пользователя, отзывает все его сессии и все его refresh token. После этого токен помечается как использованный.

В отличие от обычной смены пароля, reset password считается более чувствительной операцией, поэтому после него пользователь обычно выбрасывается из всех сессий.

## Как устроены токены

Access token не хранится в базе. Он короткоживущий, самодостаточный и валидируется по подписи. Благодаря этому API Gateway может проверять токен локально, без обращения к auth service на каждый запрос.

Refresh token, verification token и reset token хранятся в базе только в виде hash. Такой подход защищает данные: даже если база утечет, raw token злоумышленник из нее не получит.

## Как работает JWT и JWKS

JWT подписывается приватным RSA-ключом, который хранится внутри auth service и не должен попадать в репозиторий. Публичный ключ публикуется сервисом через gRPC-метод GetJWKS в формате JWKS.

API Gateway запрашивает JWKS, извлекает публичный ключ и использует его для проверки подписи JWT. После успешной проверки Gateway достает claims, например user_id, session_id и roles, и передает их дальше во внутренние сервисы через metadata/context.

## Как работает почтовый слой

Mail service нужен для отправки двух типов писем: подтверждение почты и сброс пароля. Он получает email получателя и raw token, строит frontend URL со ссылкой, формирует письмо и отправляет его через SMTP.

Логика генерации токена и хранения его hash в базе находится не в mail service, а в auth service. Это важно, потому что отправка письма — инфраструктурная задача, а создание токена и управление его жизненным циклом — часть бизнес-логики безопасности.

## Как устроены транзакции

Для простых операций достаточно обычных repository-методов. Но когда за один use-case нужно изменить несколько таблиц, используется отдельный транзакционный репозиторий.

Так сделано, например, для logout, logout all, change password и reset password. Это нужно, чтобы операции были атомарными: либо успешно изменяются и сессии, и refresh token, и пароль, либо не изменяется ничего.

## Как валидируются входные данные

Для input-моделей сервиса реализована отдельная валидация. Она проверяет обязательные поля, нормализует строки, проверяет формат email, длину пароля, наличие UUID и другие базовые ограничения.

Handler слой выполняет только transport-level проверки, например парсинг UUID из protobuf request. Основная валидация бизнес-входа происходит на уровне input-моделей и service layer.

## Что дает такой подход

В результате сервис получает понятную и устойчивую архитектуру.

Handler отвечает только за gRPC-транспорт. Service отвечает за логику. Repository отвечает за данные. Mail service отвечает за отправку писем. TXRepo отвечает за атомарные сценарии безопасности.

Благодаря этому сервис легко расширять, тестировать и сопровождать. Он может выступать как единая точка аутентификации для всей микросервисной системы, не смешивая инфраструктурные задачи с прикладной бизнес-логикой.
