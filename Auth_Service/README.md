# Auth Service

Сервис аутентификации и управления сессиями для микросервисной системы.  
Сервис реализован на Go, использует gRPC для внутренних вызовов, PostgreSQL для хранения данных и JWT + JWKS для аутентификации через API Gateway.

## Назначение сервиса

`Auth Service` отвечает за:

- регистрацию пользователя;
- вход пользователя по email и паролю;
- выдачу `access token` и `refresh token`;
- обновление токенов по `refresh token`;
- отзыв одной сессии;
- отзыв всех сессий пользователя;
- получение auth-информации о пользователе;
- выдачу JWKS для проверки подписи JWT в API Gateway.

Сервис не занимается бизнес-логикой предметной области. Его задача — быть источником identity и токенов доступа.

---

## Роль в архитектуре

Сервис предполагается использовать в схеме:

`Client -> API Gateway -> Auth Service / Business Services`

Логика взаимодействия такая:

1. Клиент отправляет запросы на логин, refresh, logout через API Gateway.
2. Gateway прокидывает запросы в Auth Service по gRPC.
3. Auth Service проверяет данные пользователя и выдает токены.
4. Access token затем передается клиентом в Gateway.
5. Gateway валидирует JWT по JWKS.
6. Gateway извлекает claims из токена и передает их дальше во внутренние сервисы через metadata/context.

Таким образом:

- **Auth Service** подписывает JWT приватным ключом;
- **API Gateway** проверяет подпись JWT по публичному ключу из JWKS;
- **внутренние сервисы** работают уже с провалидированным identity-контекстом.

## Основные сущности

### Пользователь (`users`)
Хранит основную информацию о пользователе:

- `id`
- `email`
- `username`
- `password_hash`
- `is_active`
- `email_verified`
- `created_at`
- `updated_at`

### Сессия (`sessions`)
Хранит факт логина пользователя с конкретного устройства / клиента:

- `id`
- `user_id`
- `client_id`
- `ip`
- `user_agent`
- `is_revoked`
- `revoked_at`
- `expires_at`
- `last_seen_at`
- `created_at`

### Refresh token (`refresh_tokens`)
Хранит refresh token в хешированном виде:

- `id`
- `user_id`
- `session_id`
- `token_hash`
- `is_revoked`
- `revoked_at`
- `expires_at`
- `used_at`
- `replaced_by_token_id`
- `created_at`

### Роли (`roles`, `user_roles`)
Используются для хранения ролей пользователя:

- `roles` — справочник ролей;
- `user_roles` — связь пользователь ↔ роль.

## Токены

### Access token
Access token — это JWT, который:

- живет короткое время;
- используется для доступа к API;
- не хранится в базе;
- валидируется по подписи через JWKS.

В токен кладутся claims:

- `sub` — id пользователя;
- `sid` — id сессии;
- `roles` — список ролей;
- `exp` — срок действия;
- `iat` — время выдачи;
- `iss` — кто выдал токен;
- `aud` — для кого выдан токен.

### Refresh token
Refresh token:

- живет дольше access token;
- хранится у клиента в raw-виде;
- в базе хранится только его `hash`;
- используется для получения новой пары токенов;
- ротируется при каждом refresh.

---

## Почему используется JWT + JWKS

Сервис использует асимметричную подпись JWT (`RS256`):

- приватный ключ хранится в Auth Service;
- публичный ключ выдается через JWKS;
- Gateway использует публичный ключ для проверки подписи.

Это позволяет:

- не обращаться в Auth Service на каждый запрос;
- валидировать токен локально в Gateway;
- снизить связанность между сервисами.

## Основные сценарии работы

## 1. Регистрация пользователя

### Что происходит
1. Клиент отправляет email, username и пароль.
2. Auth Service проверяет, что пользователь с таким email еще не существует.
3. Пароль хешируется через `bcrypt`.
4. В таблице `users` создается запись пользователя.
5. Возвращается `user_id`, `email`, `email_verified`.

### Важно
- при регистрации сессия не создается;
- токены не выдаются;
- роль может назначаться отдельно или автоматически.

---

## 2. Логин пользователя

### Что происходит
1. Клиент отправляет email и пароль.
2. Gateway дополнительно передает:
    - `client_id`
    - `ip`
    - `user_agent`
3. Auth Service ищет пользователя по email.
4. Проверяет пароль через `bcrypt.CompareHashAndPassword`.
5. Проверяет, что пользователь активен.
6. Создает новую запись в `sessions`.
7. Получает роли пользователя.
8. Генерирует `access token`.
9. Генерирует `refresh token`.
10. Хеш refresh token сохраняется в `refresh_tokens`.
11. Клиент получает:
    - `access_token`
    - `refresh_token`
    - время истечения токенов
    - `session_id`
    - `token_type = Bearer`

---

## 3. Refresh токенов

### Что происходит
1. Клиент отправляет `refresh token`.
2. Auth Service хеширует его и ищет в таблице `refresh_tokens`.
3. Проверяет:
    - токен существует;
    - токен не истек;
    - токен не был заменен;
    - связанная сессия существует;
    - сессия не отозвана;
    - сессия не истекла.
4. Получает роли пользователя.
5. Генерирует новый `access token`.
6. Генерирует новый `refresh token`.
7. В транзакции:
    - старый refresh token помечается как `used` и `revoked`;
    - создается новый refresh token;
    - старый токен получает ссылку `replaced_by_token_id`.
8. Клиент получает новую пару токенов.

### Почему это важно
Такой подход называется **refresh token rotation**.  
Он уменьшает риск повторного использования украденного refresh token.

---

## 4. Logout одной сессии

### Что происходит
1. Клиент передает `user_id` и `session_id`.
2. Auth Service проверяет, что сессия существует.
3. Сессия отзывается:
    - `is_revoked = true`
    - `revoked_at = now()`
4. Все refresh token этой сессии тоже отзываются.

### Важно
Сессия не удаляется физически из базы.  
Используется soft revoke, чтобы сохранять историю.

---

## 5. Logout всех сессий пользователя

### Что происходит
1. Клиент передает `user_id`.
2. Auth Service проверяет существование пользователя.
3. Все сессии пользователя отзываются.
4. Все refresh token пользователя отзываются.
5. Возвращается количество отозванных сессий.

---

## 6. Получение auth-информации о пользователе

Метод нужен для внутренних сервисов.

Возвращает:
- `user_id`
- `email`
- `roles`
- `permissions`
- `is_active`
- `email_verified`

На текущем этапе permissions могут быть пустыми.

---

## 7. Получение JWKS

Gateway вызывает метод `GetJWKS`, чтобы получить публичный ключ(и) Auth Service.

Возвращается JSON формата JWKS, который затем может быть опубликован Gateway наружу, например как:

`/.well-known/jwks.json`

---

## Почему access token не хранится в БД

Access token не хранится в базе, потому что:

- это короткоживущий JWT;
- его можно проверить по подписи;
- он самодостаточен;
- хранить его отдельно в БД обычно не нужно.

В базе хранятся:
- пользователи;
- сессии;
- refresh token в виде hash.

---

## Почему refresh token хранится как hash

Refresh token — долгоживущий секрет.  
Если БД утечет, raw token нельзя раскрывать.

Поэтому:
- клиент хранит raw refresh token;
- в БД хранится только его hash.

При refresh:
1. пришедший raw token хешируется;
2. хеш сравнивается с записью в БД.

## gRPC API

Сервис предоставляет gRPC методы:

- `Register`
- `Login`
- `Refresh`
- `Logout`
- `LogoutAll`
- `GetUserAuthInfo`
- `GetJWKS`

### Принцип handler-слоя
Handler-слой:
- принимает protobuf request;
- преобразует его во внутренние input-модели;
- парсит UUID;
- вызывает service-слой;
- преобразует результат обратно в protobuf response.

Handler не должен:
- работать с БД;
- генерировать JWT;
- хешировать пароли;
- управлять refresh rotation.

---

## Service layer

Service-слой содержит основную бизнес-логику:

- регистрацию пользователя;
- проверку пароля;
- создание сессии;
- генерацию JWT;
- генерацию refresh token;
- refresh rotation;
- revoke сессий и токенов;
- получение JWKS.

---

## Repository layer

Repository-слой отвечает только за доступ к данным:

- создание пользователя;
- поиск пользователя;
- создание сессии;
- получение сессии;
- revoke сессий;
- создание refresh token;
- поиск refresh token по hash;
- revoke токенов;
- ротацию refresh token;
- чтение ролей пользователя.

Repository не должен содержать бизнес-логику логина или refresh-flow.

## Ключи JWT

Сервис использует RSA-приватный ключ для подписи JWT.  
Публичный ключ получается из приватного и используется для формирования JWKS.

### Важно
Приватный ключ нельзя хранить в репозитории.

Нельзя коммитить:
- `private.pem`
- `.env`

Рекомендуется добавить в `.gitignore`:

```gitignore
.idea/
.env
keys/
*.pem
```

## Graceful shutdown

Сервис рекомендуется запускать с graceful shutdown:

- при получении `SIGINT` или `SIGTERM` сервер перестает принимать новые соединения;
- текущие запросы завершаются;
- gRPC сервер мягко останавливается;
- соединение с БД закрывается.

## Ключевые архитектурные решения

### Почему роли пока читаются из Auth Service
Auth Service выдает токены и должен понимать, какие роли пользователя класть в claims.  
Позже управление ролями можно вынести в отдельный сервис.

### Почему logout и logout all делают revoke, а не delete
Физическое удаление ломает аудит и усложняет диагностику.  
Поэтому используется мягкий отзыв сущностей.

### Почему refresh token ротируется
Это повышает безопасность и позволяет отслеживать повторное использование токена.

## Итог

Auth Service — это отдельный сервис identity и token management, который:

- аутентифицирует пользователя;
- управляет сессиями;
- выдает и обновляет токены;
- публикует JWKS;
- служит источником auth-информации для всей системы.

Это не бизнес-сервис, а инфраструктурная основа для всей дальнейшей микросервисной архитектуры.
